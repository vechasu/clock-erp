# Vechasu ERP — инструкции для Codex

## О проекте

Vechasu ERP — Flask-приложение для операционной работы интернет-магазина Tictactoy.

Основные функции:

* заказы Bitrix;
* товары и остатки МойСклад;
* складские ячейки;
* списания и оприходования;
* продажи;
* ремонты;
* настройки ERP.

Основные области проекта:

* `app/web.py` — Flask-маршруты и основная бизнес-логика;
* `app/clients/` — клиенты внешних интеграций;
* `app/templates/` — HTML, встроенные CSS и JavaScript;
* `instance/` — локальные JSON-хранилища.

Локальный путь:
`/Users/maksim/Projects/clock-erp`

Production-путь:
`/opt/clock-erp`

Основная ветка:
`main`

Production-сервис:
`clock-erp`

## Безопасная удалённая разработка

При работе непосредственно на сервере:

1. Работать только в `/root/codex-workspaces/clock-erp`; `/opt/clock-erp` во время обычной разработки не изменять.
2. Перед началом работы выполнять `git status` и проверять активную ветку и её актуальность.
3. Не изменять и не копировать `.env`, секреты и production-данные.
4. Не добавлять в Git базы данных, пользовательские загрузки, кеши и содержимое `instance/`.
5. После изменений запускать подходящие безопасные тесты и проверки.
6. Перед завершением просматривать и показывать итоговый diff.
7. Не выполнять deploy автоматически.
8. Не перезапускать `clock-erp.service` без отдельного прямого указания пользователя.
9. Осмысленный commit и push выполнять только после успешных проверок и прямого разрешения пользователя.

Эти правила дополняют общий процесс ниже и имеют приоритет для отдельной серверной рабочей копии.

## Агентный pipeline разработки

Эти правила применяются ко всем продуктовым задачам и имеют приоритет над общими Git-правилами ниже:

1. Каждая задача выполняется в отдельной ветке `agent/<краткое-название>` от актуальной `origin/main`.
2. Разработка и прямой push непосредственно в `main` запрещены.
3. Перед началом задачи проверять `git status`, активную ветку и актуальность `origin/main`.
4. Production `/opt/clock-erp` нельзя изменять во время обычной разработки.
5. Нельзя копировать или коммитить `.env`, секреты, базы данных, пользовательские загрузки и production-данные.
6. При смешанных изменениях запрещено бездумно выполнять `git add -A`; добавлять только файлы текущей задачи.
7. Перед commit обязательно проверять полный diff и отсутствие секретов и production-данных.
8. После реализации запускать все подходящие безопасные тесты и проверки.
9. При падении обязательных тестов задача не считается готовой и не публикуется как завершённая.
10. Bitrix и МойСклад во время тестирования должны быть отключены, подменены тестовыми объектами или использоваться только в явно разрешённом read-only режиме.
11. Каждый результат публикуется отдельным осмысленным commit и Draft Pull Request в `main`.
12. Автоматический merge запрещён; решение о merge принимает Максим отдельно.
13. Автоматический deploy запрещён; создание Issue, commit или Pull Request не разрешает deploy.
14. Перезапуск `clock-erp.service` разрешён только после отдельного прямого указания Максима.
15. В Pull Request указывать результаты тестов, риски, миграции, изменения конфигурации и способ отката.

7. Не изменять поведение других страниц без необходимости.
8. Сохранять существующие локальные изменения пользователя.
9. Не делать commit и push без прямого указания пользователя.
10. Не переходить к следующей функции самостоятельно после завершения текущей.

## Git-безопасность

Без прямого разрешения пользователя запрещено выполнять:

* `git reset`;
* `git restore`;
* `git checkout` для отката файлов;
* `git clean`;
* force push;
* удаление веток;
* переписывание истории.

Нельзя включать в commit изменения, которые не относятся к текущей задаче.

Push рабочей ветки `agent/**` и создание Draft Pull Request разрешены только прямой задачей пользователя на публикацию результата. Прямой push в `main` запрещён. SSH-подключение к production, обновление production и перезапуск systemd-сервиса разрешены только после точной фразы пользователя:

`ОКЕЙ, РАЗВОРАЧИВАЙ`

После этой фразы:

1. Проверить `git status`.
2. Проверить итоговый diff.
3. Убедиться, что одобренный Pull Request отдельно влит в `origin/main`.
4. Убедиться, что production-репозиторий не содержит неизвестных изменений.
5. Запустить `scripts/deploy.sh` для уже влитого commit.
6. Проверить статус сервиса и HTTP-ответ приложения.
7. Показать номер развёрнутого commit, результат развёртывания и итоговый `git status`.

## Безопасный процесс разработки и развёртывания

1. Перед началом работы выполнить `git status`, обновить сведения об `origin/main` и создать отдельную ветку `agent/<краткое-название>` от актуальной `origin/main`.
2. Не изменять файлы напрямую на production-сервере.
3. Все изменения сначала выполнять в локальном Git-репозитории `/Users/maksim/Projects/clock-erp`.
4. После изменений запускать доступные проверки Python и проверку шаблонов.
5. Перед завершением показывать краткий список изменённых файлов и результаты проверок.
6. Рабочую ветку `agent/**` публиковать только по прямой задаче пользователя и только через Draft Pull Request; не выполнять прямой push в `main`, SSH-подключение к production, обновление сервера и `systemctl restart` без точной фразы `ОКЕЙ, РАЗВОРАЧИВАЙ`.
7. После этой фразы развернуть уже отдельно влитый и одобренный commit через `scripts/deploy.sh`, проверить сервис и HTTP-ответ, затем сообщить номер commit и результат.
8. Если хотя бы одна обязательная проверка не прошла, не развёртывать изменения.
9. Если на сервере есть незакоммиченные изменения, остановить развёртывание, ничего не удалять и сообщить о блокировке.
10. Не выполнять `git reset --hard`, пока непосредственно перед этой командой не подтверждена чистота рабочей директории сервера.

## Безопасность данных и интеграций

1. Не читать и не показывать содержимое `.env`.
2. Не выводить токены, пароли, ключи, cookie и другие секреты.
3. Не выполнять реальные операции записи в Bitrix или МойСклад без прямого разрешения пользователя.
4. Не менять статусы реальных заказов во время проверки.
5. Не создавать реальные списания, оприходования или товары во время тестирования.
6. Не изменять production-сервер без прямого указания.
7. Не устанавливать зависимости без разрешения.
8. Не использовать настоящие данные из `instance/` для разрушительных тестов.
9. Не заменять повреждённые или непонятные данные пустыми значениями без явного предупреждения.

## Правила изменения кода

1. Сохранять текущую архитектуру, если задача прямо не требует рефакторинга.
2. Не добавлять новые библиотеки, когда задачу можно решить средствами проекта.
3. Не переносить большие участки кода без необходимости.
4. Не переименовывать маршруты, функции, поля форм и JSON-ключи без проверки всех зависимостей.
5. Учитывать, что шаблоны содержат встроенные CSS и JavaScript.
6. Для изменений интерфейса сохранять общий визуальный стиль Vechasu ERP.
7. Проверять отображение таблиц, форм, модальных окон и мобильной версии.
8. Не доверять данным браузерной формы при критических складских операциях без серверной проверки.
9. Не подавлять ошибки внешних API без понятного сообщения или журналирования.
10. Не утверждать, что функция проверена, если соответствующая проверка не выполнялась.

## Проверки после изменений

После каждой задачи выполнить все подходящие безопасные проверки:

* проверку синтаксиса изменённых Python-файлов;
* проверку изменённых JSON-файлов;
* `git diff --check`;
* существующие тесты проекта;
* безопасную проверку Flask-маршрутов без операций записи;
* просмотр итогового diff;
* итоговый `git status`.

Если тестов для задачи нет, прямо указать это в отчёте.

Нельзя выполнять тесты, способные изменить реальные данные Bitrix, МойСклад или production.

## Формат итогового отчёта

После завершения задачи кратко показать:

1. Что было сделано.
2. Какие файлы изменены.
3. Какие проверки выполнены.
4. Что не удалось проверить.
5. Возможные риски или ограничения.
6. Итоговый `git status`.

Отвечать пользователю на русском языке, понятно и без лишней технической воды.

Если задача сформулирована достаточно ясно, выполнить её без ненужных уточняющих вопросов. При неоднозначности выбрать минимальное и наиболее безопасное изменение.
