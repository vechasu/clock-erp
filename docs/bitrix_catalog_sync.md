# Ручная синхронизация каталога Bitrix

Команда:

```bash
venv/bin/python scripts/sync_bitrix_catalog.py
```

Команда выполняет только read-only GET к endpoint Bitrix и записывает контент в
SQLite-каталог Vechasu. Она всегда запрашивает также неактивные карточки, чтобы
корректно переносить деактивацию, но не импортирует остатки и не вызывает API
МойСклад.

## Курсор и восстановление

Для первого запуска нижняя граница берётся из максимального
`external_updated_at` уже загруженного каталога. Далее используется `cursor_to`
последнего успешного `incremental_sync`.

Каждая обработанная страница сохраняет счётчики прогресса в
`catalog_sync_runs`. Новый `cursor_to` записывается только при полном успехе.
При частичной ошибке parent-run получает статус `failed`, `cursor_to` остаётся
пустым, а повторный запуск использует прежний `cursor_from`. Уже записанные
страницы безопасно обрабатываются повторно по `payload_hash` и уникальному
Bitrix ID.

Критическая ошибка завершает CLI ненулевым кодом. В журнал сохраняется только
тип исключения без URL, токена или текста ответа.

## Реальная проверка 20.07.2026

Первый ручной запуск:

* cursor: `2026-07-20T10:59:30+00:00` → `2026-07-20T12:45:55+00:00`;
* страниц: 1;
* получено: 1;
* без изменений: 1;
* создано/обновлено/конфликты: 0/0/0.

Повторный ручной запуск:

* cursor: `2026-07-20T12:45:55+00:00` → `2026-07-20T12:46:02+00:00`;
* страниц: 1;
* получено: 0;
* создано/обновлено/конфликты: 0/0/0.

Третий ручной запуск после аудита свойств и обновления интерфейса:

* cursor: `2026-07-20T12:46:02+00:00` → `2026-07-20T13:17:00+00:00`;
* страниц: 1;
* получено: 1;
* обновлено: 1;
* карточка Bitrix `236966` действительно перешла из активных в архивные;
* создано/без изменений/конфликты: 0/0/0.

Контрольный полный проход `--verify-all` после этого запуска:

* получено 4684 карточки на 24 страницах;
* без изменений: 4684;
* создано/обновлено/конфликты: 0/0/0;
* incremental cursor остался `2026-07-20T13:17:00+00:00`;
* `cursor_advanced=false`, складских операций и записей в МойСклад — 0.

Заключительный инкрементальный запуск:

* cursor: `2026-07-20T13:17:00+00:00` → `2026-07-20T13:20:58+00:00`;
* получена 1 граничная карточка, результат `unchanged`;
* создано/обновлено/конфликты: 0/0/0;
* курсор успешно продвинут.

После запусков в каталоге осталось 4684 товара и 0 дублей. Все перечисленные
parent-run успешны, неуспешных incremental-run нет. `PRAGMA quick_check` вернул `ok`,
ошибок внешних ключей нет. Хэши файлов `instance`, не относящихся к каталогу,
совпали с исходными; read-only проверка МойСклад снова вернула `meta.size = 0`.

Автоматическое расписание, cron и systemd timer не создавались.

Известное ограничение Bitrix: `updated_from` опирается на `TIMESTAMP_X`
элемента. Изменение только строки цены может не менять эту дату, поэтому для
контроля цен нужен периодический ручной полный проход. Он также не должен
импортировать остатки.

## Проверки обнаружения изменений

Без изменения реальных карточек тестами подтверждено, что импортёр обнаруживает
изменение цены, свойства и добавление изображения. Отдельные интеграционные
сценарии подтверждают, что ошибка одной страницы не продвигает parent cursor, а
повторная обработка уже сохранённой страницы не создаёт дублей. Реальная
production-деактивация карточки дополнительно подтвердила обработку изменения
Bitrix между ручными запусками.

Реальные цену, свойство и изображение в Bitrix специально не меняли. Поэтому их
production-обнаружение подтверждено контролируемыми тестами, а не разрушительным
экспериментом с данными магазина.

## Рекомендации по расписанию

Предпочтителен отдельный systemd oneshot-service и timer, а не cron: так проще
запретить параллельные запуски, увидеть exit code и журнал, а также выполнить
пропущенный запуск после перезагрузки (`Persistent=true`). Рекомендуемый режим:

* обычный incremental sync каждые 15–30 минут;
* защита от одновременного запуска через `flock`;
* `RandomizedDelaySec` в несколько минут, чтобы не создавать точный пик запросов;
* полный `--verify-all` раз в сутки ночью или хотя бы раз в неделю для контроля
  ценовых изменений, не отражённых в `TIMESTAMP_X`;
* уведомление при ненулевом exit code и контроль возраста последнего успешного
  `catalog_sync_runs`;
* резервная копия `catalog.db` по отдельной политике ротации.

Перед включением timer нужно сначала отдельно согласовать частоту, пользователя
сервиса, путь к журналу и хранение резервных копий. В рамках текущей задачи
unit-файлы и cron-записи не создавались.
