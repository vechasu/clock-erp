## Аудит 56 свойств Bitrix

Проверены определения и реальные значения всех 56 свойств. Надёжного свойства
`MODEL`, `MODEL_CODE`, `ARTICLE`, `VENDOR_CODE`, `MANUFACTURER_CODE`, `SKU` или
`EXTERNAL_CODE` в снимке нет. Поле `RETARGETING` (`Код ретаргетинга`) не
заполнено ни у одного товара. `BRAND_MODEL` (`Марка часов`) заполнено у 3 804
товаров и содержит 67 брендов, поэтому используется только для проверки бренда,
а не как модель.

Похожие на коды фрагменты в `PROP_MECHANISM`, `PROP_MATERIAL`, `VIDEO_LINK`,
`GUARANTY` и других полях являются механизмами, характеристиками, URL или
служебным текстом. Они смешаны с обычными значениями и исключены из модельного
индекса. Цветовые и размерные свойства пригодны только для обнаружения
конфликта варианта. Вне набора свойств у всех 4 684 товаров есть уникальные
`external_xml_id` и Bitrix ID, но Excel не содержит соответствующих полей.

Воспроизводимый аудит выполняет
`scripts/audit_bitrix_matching_fields.py`; он создаёт JSON, CSV и Markdown с ID,
кодом, названием, типом, числом заполненных товаров, уникальными значениями,
примерами и оценкой пригодности для каждого свойства.

## Консервативный алгоритм

Инструмент `scripts/reconcile_bitrix_excel_catalog.py` открывает SQLite только в
режиме `mode=ro&immutable=1`, читает Excel и создаёт JSON/CSV dry-run. Он не
импортирует Flask-приложение, внешние клиенты или функции прихода.

Индексы строятся по `XML_ID`, ID Bitrix, безопасному коду модели из разрешённого
свойства, коду модели из названия/артикула, бренду + названию, названию без
ведущего бренда и строгому набору значимых токенов.

Автоматические статусы ограничены двумя случаями:

1. `exact` — единственная карточка с тем же нормализованным названием внутри
   того же нормализованного бренда;
2. `high_confidence` — единственная карточка по точному коду модели, названию
   без ведущего бренда или строгому одинаковому набору значимых токенов.

Извлечение модели сохраняет исходное и нормализованное значение и поддерживает
формы `RA-AC0M04Y`, `RA AC0M04Y`, `RAAC0M04Y`, `PG-3`, `PG03`, `X7000GA-Y`,
слеши и дефисы. `PG-3` и `PG03` считаются одной доказанной формой; широкое
удаление ведущих нулей для других кодов не применяется. Общие слова `BLACK`,
`GOLD`, `MOON`, `ECLIPSE`, размеры и URL кодами модели не считаются.

Нормализация названия меняет регистр, Unicode-дефисы, повторные пробелы, слеши,
амперсанды и знаки пунктуации. Доказанные эквиваленты ограничены формами
`rose gold`/`rosegold` и `gun metal`/`gunmetal`. Fuzzy-поиск остаётся только
ручным кандидатом.

Автоматическое решение запрещается при конфликте бренда, кода, цвета, размера,
образца/комплекта, общем однословном названии (`Black`, `Gold`, `Moon`,
`Eclipse` и аналоги) или нескольких кандидатах. Отсутствующий бренд допускается
только при уникальном коде без иных конфликтов. Несколько Excel-строк разных
цветов не могут автоматически использовать одну карточку Bitrix.

Идентичность внутренней складской карточки определяется номером строки Excel,
а не нормализованным названием или артикулом. Поэтому одинаковые названия не
объединяются, остатки не суммируются, а исходные поля и различия каждой строки
сохраняются отдельно. Если несколько строк безопасно сопоставлены одной
карточке Bitrix, связь явно получает тип `many_to_one`: контент можно показать
у каждой складской карточки, но остатки остаются независимыми. Совпадение лишь
ручных кандидатов помечается `many_to_one_candidate` и контент не подключает.

## Реализованная внутренняя модель

Excel является единственным источником состава, фактического остатка и ячейки.
Bitrix используется только как read-only источник снимка контента при
подтверждённой связи. МойСклад не участвует в применении batch; у карточек и
партии сохраняется `moysklad_sync_status = not_linked`.

В `catalog.db` добавлены пять внутренних сущностей:

1. `catalog_excel_batches` — уникальный SHA-256, стабильный `batch_id`, источник,
   контрольные суммы строк и остатков, предшествующая партия и статус;
2. `catalog_excel_products` — одна активная карточка на устойчивый ключ номера
   строки Excel, исходные поля, раздельный остаток и отдельный снимок обогащения
   Bitrix, включая явную cardinality связи;
3. `catalog_excel_batch_rows` — старое и применённое состояния каждой строки;
4. `catalog_excel_stock_operations` — корректировки «старое → новое» и обратные
   операции rollback только для ненулевой разницы;
5. `catalog_excel_match_audit` — ручная привязка, отметка «В Bitrix нет», отмена
   связи и undo.

Apply выполняется одной SQLite-транзакцией. Для каждой строки рассчитывается
`difference = excel_stock - old_stock`, после чего внутренний остаток
устанавливается ровно в `excel_stock`. Поэтому 0 — полноценное целевое значение,
но движение с разницей 0 не создаётся. Для первой загрузки планируется 836
ненулевых корректировок; dry-run не применяет их, поэтому фактических движений
0. Повтор того же SHA-256 возвращает существующую партию без новых
операций.
Rollback разрешён только для активной партии: он восстанавливает сохранённые
поля и остатки предыдущей партии, удаляет только созданные партией карточки без
ручной истории или использования другими партиями и пишет обратные операции
только для ненулевых возвратных корректировок.

Команда `scripts/apply_excel_product_catalog.py` по умолчанию выполняет только
preview. Для apply одновременно нужны `--apply` и точный
`--confirm-file-sha`. Одинаковые названия или общая карточка Bitrix не блокируют
batch; только `invalid` останавливает всю транзакцию до записи. Для отката
используется `--rollback-batch <batch_id>`.

Интерфейс `/products` по умолчанию показывает только активный Excel batch.
Bitrix-only карточки доступны отдельно через `/catalog`. Фильтр «Требует
сопоставления» показывает `ambiguous`, а карточка товара позволяет выбрать
кандидата, отметить отсутствие в Bitrix, отменить связь или вернуть последнее
ручное решение. Подтверждение меняет только локальное обогащение; Excel-остаток
и ячейка не изменяются.

## Проверки безопасности

- dry-run дважды даёт детерминированный результат;
- `catalog.db` открыт read-only, его checksum не меняется;
- инструмент не импортирует и не вызывает Bitrix/MoySklad-клиенты;
- применение и rollback покрыты транзакционными unit-тестами на временной БД;
- проверено сохранение 3 313 уникальных строк без потери;
- выполнено 0 POST/PUT/DELETE-запросов;
- создано 0 товаров, приходов и складских операций;
- в рамках этого follow-up batch не применялся повторно, сервис не
  перезапускался, deploy и merge не выполнялись;
- все 158 новых связей по модели и выборка более 50 новых автоматических связей
  проверены: точный код присутствует с обеих сторон, бренд совместим, конфликтов
  цвета/размера/комплекта нет;
- проверены все 26 групп одинаковой базовой модели с разными цветами: 0 случаев,
  когда разные цвета автоматически связаны с одной карточкой Bitrix;
- единственная автоматическая группа `many_to_one` осталась прежней: строки
  2380 и 2383, раздельные остатки 3 и 0;
- общие названия `Black`, `Gold`, `Eclipse`, витринные образцы и комментарии не
  получили новых автоматических связей.
