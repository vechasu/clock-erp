## Консервативный алгоритм

Инструмент `scripts/reconcile_bitrix_excel_catalog.py` открывает SQLite только в
режиме `mode=ro&immutable=1`, читает Excel и создаёт JSON/CSV dry-run. Он не
импортирует Flask-приложение, внешние клиенты или функции прихода.

Индексы строятся по ID Bitrix, `XML_ID`, артикулу, точному нормализованному
названию, бренду + названию и бренду + названию без ведущего бренда. Для
исследуемого Excel первые три индекса не дают совпадений из-за отсутствия ID в
Excel и SKU в Bitrix.

Автоматические статусы ограничены двумя случаями:

1. `exact` — единственная карточка с тем же нормализованным названием внутри
   того же нормализованного бренда;
2. `high_confidence` — то же условие после безопасного удаления бренда из
   начала названия Excel.

Нормализация меняет регистр, Unicode-дефисы, повторные пробелы, слеши,
амперсанды и знаки пунктуации. Цвета, размеры и значимые слова не удаляются.
Поиск похожих названий и токенов модели используется только для списка ручных
кандидатов и никогда не даёт автоматический статус.

Автоматическое решение запрещается при различии бренда, общем однословном
названии (`Black`, `Gold`, `Moon`, `Eclipse` и аналоги), нескольких карточках,
различии признаков образца/комплекта и конфликте нескольких строк Excel с одной
карточкой. Различия цвета и размера остаются в названии, поэтому похожие
варианты попадают только на ручную проверку.

## Безопасный план объединения

Реальный import/apply в эту задачу намеренно не входит. Перед ним требуется
отдельное согласование и минимальная транзакционная модель:

1. Максим вручную подтверждает 1 004 спорные строки, 917 отсутствующих строк,
   2 группы дублей и допустимые алиасы брендов. `not_found` не создаются
   автоматически.
2. Для каждой подтверждённой строки сохраняется mapping: SHA-256 файла,
   `batch_id`, номер строки, исходное название/артикул/ячейка, ID и `XML_ID`
   карточки Bitrix, подтверждённый ID товара МойСклад и статус решения.
3. На уровне batch создаётся уникальное ограничение по SHA-256 файла. Тот же
   batch нельзя применить повторно. Текущий dry-run формирует стабильный
   `batch_id` и умеет отклонить уже известный ID, но production-реестр ещё не
   реализован — это дополнительная причина не проводить импорт сейчас.
4. Перед apply сохраняются старые складские поля и остатки, ID каждого
   созданного документа прихода и локальной записи операции. Ячейка, валидный
   складской артикул и исходное Excel-название записываются только в слой
   складского mapping/alias, не затирая контент Bitrix.
5. Для строк с положительным остатком создаётся один проверяемый приход только
   по подтверждённым ID МойСклад. Для нулевых строк сохраняется mapping без
   движения количества.
6. Откат одного batch удаляет его документ прихода, связанные локальные
   операции и его mapping/alias-изменения, восстанавливает предыдущие поля и не
   касается других документов. Новые товары нельзя удалять автоматически, если
   после batch они участвовали в других операциях.

До появления подтверждённых связей Bitrix→МойСклад, production batch-реестра,
уникального ограничения SHA-256 и проверенного отката существующий endpoint
`/receipts/create` использовать для этого файла нельзя.
