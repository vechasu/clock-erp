# Исследование импорта заказов Битрикса

Дата проверки: 2026-07-20. На этом этапе выполнены только GET-запросы. Записей в
Bitrix, МойСклад, `instance/` и production не выполнялось.

## Фактическая интеграция

Проект получает список и карточки заказов через два настраиваемых read-only
endpoint: `BITRIX_ORDERS_URL` и `BITRIX_ORDER_URL`. Production-адреса намеренно
не хранятся в Git. Формат статусов и структура ответа указывают на модуль `sale`
интернет-магазина «1С-Битрикс: Управление сайтом». Это не CRM-сделки, CRM-заказы
и не смарт-процессы Битрикс24.

Сейчас Vechasu не хранит полноценные заказы: страница читает их из Bitrix в
оперативный кеш. В локальных JSON есть только связанные сущности:

* `product_mappings.json`: внутренний ID товара Bitrix → ID товара МойСклад;
* `stock_operations.json`: `order_id`, `order_number`, ID/название товара Bitrix
  после фактического списания;
* `manual_sales.json`: номер заказа и отдельная строка продажи, без внешнего ID
  заказа и без гарантированной связи с Bitrix;
* `automatic_sales_overrides.json`: правки строк продаж по ID складской операции.

## Результат ограниченного read-only исследования

Исследованный list endpoint может возвращать фиксированное окно и игнорировать
меньший `limit`. Поэтому клиент всегда дополнительно ограничивает результат на
своей стороне максимум десятью карточками. Конкретные production ID, номера
заказов и значения полей в Git не сохраняются.

В ограниченной выборке подтверждено наличие идентификатора и номера заказа, даты
создания, статуса, признака оплаты, итоговой суммы, валюты и товарных строк.
Клиент также распознаёт наличие полей клиента, но dry-run не сохраняет данные и
по умолчанию не печатает подробный preview.

Пользовательское свойство `FROM` является маркетинговой атрибуцией, а не
техническим каналом/trade platform. В исследованном контракте не подтверждены:
дата изменения, технический источник, название платёжной системы, доставка,
комментарий, XML_ID, SKU/артикул, исходная цена, скидка, готовая сумма строки и
закупочная цена. Сумма строки в dry-run только вычисляется как фактическая цена
единицы × количество и явно помечена как вычисленная.

Поле `products[].price` — фактическая цена продажи, которую текущий API отдаёт
для позиции корзины. Отдельного поля исходной цены нет, поэтому размер скидки
доказуемо восстановить нельзя. Закупочная цена отсутствует и ценой продажи не
подменяется.

Read-only исследование показало, что большинство строк без расширенного
контракта требуют ручного сопоставления. Автоматическое сопоставление применяет
XML_ID, затем SKU, затем уникальное точное название. Неоднозначные названия не
принимаются.

## Сопоставление полей

| Поле Bitrix | Поле Vechasu | Преобразование | Обязательность |
|---|---|---|---|
| `ID` | `external_id` | строка; уникально вместе с `external_source` | обязательно |
| константа | `external_source` | `bitrix` | обязательно |
| `ACCOUNT_NUMBER` / `number` | `order_number` | строка, не использовать как ключ дедупликации | обязательно |
| `DATE_INSERT` / `date` | `created_at` | ISO-8601 с часовым поясом | обязательно |
| `DATE_UPDATE` | `external_updated_at` | ISO-8601; нужен расширенный API | обязательно для обновлений |
| время успешной синхронизации | `last_synced_at` | UTC ISO-8601 | обязательно |
| `STATUS_ID` | `status_external` | хранить исходный код и отдельное отображаемое имя | обязательно |
| `FIO` | `customer_name` | значение свойства заказа | желательно |
| `PHONE` | `customer_phone` | нормализовать, исходное значение сохранить | желательно |
| `EMAIL` | `customer_email` | trim/lowercase для поиска, исходное сохранить | желательно |
| trade platform / affiliate | `source` | таблица кодов источников | желательно |
| payment collection / `PAYED` | `payment_*` | отдельно способ, сумма, статус и дата | желательно |
| shipment collection | `delivery_*` | отдельно служба, стоимость, статус, трек | желательно |
| `USER_DESCRIPTION` / `COMMENTS` | `comment` | строка без HTML | необязательно |
| `PRICE` заказа | `total_amount` | Decimal, не float в постоянном хранилище | обязательно |
| `CURRENCY` | `currency` | ISO-код валюты | обязательно |
| basket `ID` | `external_line_id` | уникально внутри источника | обязательно |
| basket `PRODUCT_ID` | `external_product_id` | строка; не равно XML_ID | желательно |
| catalog/basket `XML_ID` | `product_xml_id` | trim, точное сравнение | желательно |
| catalog `article`/`code` | `product_sku` | trim/casefold для индекса | желательно |
| basket `NAME` | `product_name_external` | исходное название | обязательно |
| basket `QUANTITY` | `quantity` | Decimal > 0 | обязательно |
| basket `BASE_PRICE` | `original_unit_price` | Decimal; текущий API не отдаёт | желательно |
| basket `PRICE` | `sale_unit_price` | Decimal; цена после скидок/наценок | обязательно |
| разница цен/discount data | `discount_amount` | не вычислять без исходной цены | желательно |
| basket total или `PRICE*QUANTITY` | `line_total` | Decimal; хранить признак вычисления | обязательно |
| catalog purchase price | `purchase_unit_price` | только отдельное явное поле и право | необязательно |

## Требуемое постоянное хранилище

Текущие JSON продаж и складских операций не являются базой заказов. Для полной
истории предпочтительна SQLite (есть в стандартной библиотеке Python) с таблицами:

1. `external_orders`: уникальный индекс `(external_source, external_id)`, номер,
   даты Bitrix, статус, клиент, оплата, доставка, суммы, валюта, комментарий,
   `payload_hash`, `last_synced_at`.
2. `external_order_items`: FK заказа, внешний ID строки и товара, XML_ID, SKU,
   исходное название, количество и все три раздельные цены.
3. `external_product_mappings`: источник, внешний XML_ID/ID/SKU, товар МойСклад,
   метод сопоставления и признак ручного подтверждения.
4. `external_sync_runs`: период, курсор, счётчики, ошибки, время начала/окончания.

Повторный импорт делает upsert только по `(external_source, external_id)`. Если
`DATE_UPDATE` и `payload_hash` не изменились — это дубль без записи. Если изменились
— обновляются поля заказа и строки одной транзакцией, но складские движения не
создаются автоматически. Удалённые из ответа позиции нельзя удалять без полного
снимка карточки. `last_synced_at` меняется только после успешной транзакции.

## API и массовая выгрузка

Для штатного REST подходят `sale.order.list` (страница 50), затем пакетные
`sale.basketitem.list`, методы свойств заказа, оплат и отгрузок. Альтернатива для
этого сайта — расширить существующий read-only PHP API на D7
`Bitrix\\Sale\\Order::getList`, обязательно добавив фильтры периода, `limit`,
`offset`/курсор, `total`/`next` и все коллекции карточки.

Если list endpoint отдаёт только фиксированное окно последних заказов, выгрузить
через него всю историю нельзя. REST-схема «страницы заказов + отдельные карточки»
для `N` заказов потребует примерно `ceil(N/50) + N` запросов. Пакетная выдача
корзины, свойств, оплат и доставок сокращает число запросов; лучший контракт
собственного endpoint возвращает полный пакет страницы и требует примерно
`ceil(N/50)` запросов.

Клиент должен обрабатывать 401/403 без повтора, 429 и 5xx с `Retry-After` и
экспоненциальной задержкой, сетевые timeout с ограниченным повтором, сохранять
курсор после каждой успешно записанной страницы и уметь продолжать запуск.

## Концепция интерфейса

На странице заказов: кнопка «Импорт из Битрикса» открывает мастер с периодом и
кнопкой «Предварительная проверка». Результат проверки показывает найдено / новые /
обновляемые / дубли / ошибочные, отдельную таблицу «Требует сопоставления» и
предпросмотр заказов и цен. Запись доступна только после подтверждения, отдельным
действием; итог показывает счётчики, ошибки и ID запуска. Dry-run никогда не
меняет заказы, товары, клиентов, остатки или складские операции.

Текущий диагностический скрипт не загружает `.env` автоматически и не выполняет
HTTP-запросы без явного флага. Безопасный запуск только со сводным результатом:

```bash
BITRIX_ORDERS_URL=https://example.test/orders \
BITRIX_ORDER_URL=https://example.test/order \
python scripts/bitrix_orders_dry_run.py \
  --limit 10 \
  --no-catalog \
  --allow-read-only-network
```

Подробный preview включается отдельным `--include-preview` и может содержать
реальные номера заказов и товарные данные, поэтому его нельзя сохранять в Git,
публиковать в PR или пересылать в чат. Для read-only сопоставления с МойСклад
нужно убрать `--no-catalog` и заранее передать `MOYSKLAD_TOKEN` только через
окружение; скрипт вызывает исключительно метод чтения каталога.

## Доступ

Endpoint с заказами и персональными данными должен быть закрыт авторизацией,
ограничением сети и rate limit. Отсутствие авторизации является блокирующим
риском и не должно фиксироваться в публичной документации как свойство
конкретного production-адреса.

Клиент принимает только HTTPS endpoint. Если защищённый endpoint использует
Bearer-аутентификацию, значение передаётся через `BITRIX_ORDERS_TOKEN` в
окружении процесса и не сохраняется в `.env`, Git, логах или отчёте dry-run.

Для полного REST-доступа владелец сайта должен в административной части именно
«1С-Битрикс: Управление сайтом» установить/использовать модуль REST и создать
отдельный входящий webhook или локальное приложение с минимальным scope `sale`
на чтение. Scope `catalog` нужен только для XML_ID/SKU и отдельной закупочной цены;
его не следует выдавать, если эти поля вернёт защищённый серверный endpoint.
Секрет передаётся владельцем только через защищённое хранилище и окружение
процесса, не через `.env`, чат, код или Git. Если REST-модуль не используется,
разработчик сайта должен защитить endpoint отдельным read-only сервисным ключом
и передать URL через переменные окружения.
