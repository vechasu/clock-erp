# Vechasu Agent Pipeline

## Назначение

Agent Pipeline отделяет подготовку задачи, реализацию, проверку и выпуск в production. Создание Issue, ветки, commit или Pull Request никогда не означает разрешение на merge или deploy.

Основной процесс:

1. Максим выбирает продуктовую задачу и определяет ожидаемый результат.
2. ChatGPT превращает идею в техническое задание и проверяемые критерии готовности.
3. Задача сохраняется в GitHub Issue либо передаётся Codex с тем же набором ограничений.
4. Codex обновляет сведения об `origin/main` и создаёт ветку `agent/<название>`.
5. Codex изучает существующую реализацию и вносит минимальное изменение.
6. Codex запускает безопасные тесты и проверяет полный diff.
7. Codex создаёт один осмысленный commit и отправляет только рабочую ветку.
8. Codex создаёт Draft Pull Request в `main`.
9. ChatGPT читает diff и результаты тестов, оценивает архитектуру, риски и критерии готовности.
10. Codex исправляет замечания в той же ветке и повторяет проверки.
11. Максим принимает продуктовое решение о готовности изменения.
12. Merge выполняется отдельно после проверки и решения Максима.
13. Deploy выполняется отдельно только после прямого разрешения.
14. После deploy проверяется приложение; при проблемах применяется заранее описанный откат.

Автоматический merge и автоматический deploy запрещены.

## Роли

- **Максим** — определяет направление и принимает продуктовые решения.
- **ChatGPT** — формулирует задачи, анализирует архитектуру и проверяет Pull Request.
- **Codex** — реализует изменения, запускает тесты и исправляет замечания.
- **GitHub** — хранит задачи, код, тесты и историю решений.
- **Production** — изменяется только после отдельного прямого одобрения.

## Ветки и commit

- Каждая задача начинается от актуальной `origin/main` в ветке `agent/<краткое-название>`.
- Название содержит только понятные латинские слова, цифры и дефисы, например `agent/fix-order-filter`.
- Разработка и прямой push в `main` запрещены.
- Один commit должен описывать один завершённый этап и не включать посторонние изменения.
- При смешанной рабочей директории файлы добавляются явно; `git add -A` без проверки запрещён.
- Перед commit проверяются `git status`, полный `git diff`, `git diff --check` и отсутствие секретов и production-данных.

## Готовность Draft Pull Request

Draft PR готов к проверке, когда:

- критерии задачи реализованы в пределах согласованной области;
- обязательные тесты прошли;
- полный diff проверен автором;
- PR описывает изменения, затронутые процессы и ручную проверку;
- указаны риски, миграции, конфигурация и инструкция отката;
- в ветке нет `.env`, секретов, баз данных, пользовательских загрузок и production-данных;
- тесты не записывают данные в Bitrix или МойСклад;
- PR не включает merge или deploy.

Падение обязательного теста означает, что результат не готов. Ошибка должна быть исправлена либо явно вынесена на решение Максима до публикации результата как готового.

## Исправление замечаний

ChatGPT и Максим оставляют замечания в Draft PR. Codex проверяет каждое замечание по коду, исправляет подтверждённые проблемы в той же ветке, повторяет тесты и добавляет новый осмысленный commit. История обсуждения и проверок сохраняется в PR.

## Merge

Merge выполняется только после успешных проверок, устранения обязательных замечаний и продуктового решения Максима. Автоматическое принятие решения и автоматический merge запрещены. Перед merge повторно проверяются актуальность ветки, GitHub Actions, состав diff и план миграции.

## Deploy и откат

Deploy является отдельной операцией после merge и требует прямого разрешения Максима. До разрешения нельзя подключаться к production, изменять `/opt/clock-erp`, запускать `scripts/deploy.sh` или перезапускать `clock-erp.service`.

Перед deploy фиксируются версия, необходимые миграции, способ проверки и способ отката. После deploy проверяются сервис и HTTP-ответ приложения. При проблеме откат выполняется к заранее известной рабочей версии без удаления production-данных и без переписывания Git-истории.

## Операции с обязательным участием человека

- утверждение продуктового результата;
- решение о merge;
- разрешение на миграцию production-данных;
- разрешение на deploy;
- перезапуск production-сервиса;
- выбор способа отката при риске потери данных;
- предоставление или изменение секретов и прав внешних интеграций.

## Данные и интеграции

Запрещено копировать в рабочую среду или GitHub production `.env`, токены, пароли, ключи, базы данных, пользовательские загрузки, персональные данные клиентов и содержимое production `instance/`. Bitrix и МойСклад в тестах отключаются или заменяются fake/mock-клиентами; read-only диагностика допускается только при явном разрешении.

## Среды

- **Локальная или облачная разработка** использует отдельную ветку, тестовые данные, временные базы и безопасные проверки. Она может создавать Issue, commit, push рабочей ветки и Draft PR.
- **GitHub Actions** читает код, устанавливает публичные зависимости и запускает unit-тесты без production secrets, SSH и deploy-прав.
- **Production** находится в `/opt/clock-erp`, содержит реальные настройки и данные и не является рабочей директорией для разработки. Его изменение всегда выполняется отдельным контролируемым этапом.
